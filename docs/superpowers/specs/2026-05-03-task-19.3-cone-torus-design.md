# Task 19.3 Design Spec: Analytical Primitives — Cone and Torus

**Date:** 2026-05-03  
**Sprint:** 19  
**Status:** Approved

---

## Goal

Add two new GPU-side analytical primitives — a **capped cone** and a **Y-axis torus** — using the
params-indirection IS pattern established by the cylinder primitive.

---

## Pattern

Both primitives follow the **params-indirection pattern** documented in
`docs/dev/adding-analytical-primitives.md`:

- SBT hit-group records carry no per-instance data.
- The IS program reads `optixGetInstanceId()`, looks up
  `params.instance_materials[instanceId].texture_index`, and uses that to index into
  `params.cone_data[]` or `params.torus_data[]`.
- Parameter updates require only a `cudaMemcpy`; no SBT rebuild.

This is identical to how the cylinder works and scales to any number of instances.

## Cone Shader (`hit_cone.cu`)

Template: copy `hit_cylinder.cu`.

### Intersection (`__intersection__cone`)

Standard infinite-cone ray intersection (quadratic in `cos²(half_angle)` form), then clamp
`proj ∈ [0, height]` to obtain the finite lateral surface. Additionally test the **base disk cap**
(`proj == height`, i.e. the plane at distance `height` from apex along `axis`):

```
disk test: dot(P - base, axis) ≈ 0  AND  length(P - base - (dot(P-base,axis))·axis) ≤ radius
```

Both tests use the standard `[tmin, tmax]` guard and `optixReportIntersection`.

### Normal

- **Lateral surface:** `N = normalize(radial · cos_a − axis · sin_a)` where
  `radial = normalize((P − apex) − proj · axis)`.
- **Base cap:** `N = −axis` (pointing away from apex).

### Entry points required

| Entry point | Purpose |
|---|---|
| `__intersection__cone` | all three hit groups |
| `__closesthit__cone` | radiance |
| `__closesthit__cone_shadow` | opaque shadow |
| `__anyhit__cone_shadow` | transparent shadow accumulation |

Closest-hit programs: copy cylinder's PBR shading verbatim; only normal computation differs.

## Torus Shader (`hit_torus.cu`)

### Intersection (`__intersection__torus`)

Y-axis torus only (Sprint 19). Quartic solved via **Schwarze/Ferrari-Cardano**:

1. Translate ray so torus center is at origin.
2. Form the standard torus quartic coefficients from `(x²+y²+z²+R²−r²)² − 4R²(x²+z²) = 0`
   substituting `x = ox + t·dx`, etc.
3. Solve `t⁴ + c3·t³ + c2·t² + c1·t + c0 = 0` using the `__device__ int solveQuartic(...)` helper.
4. Take the smallest positive real root in `[tmin, tmax]`.

`solveQuartic` signature:
```cpp
__device__ int solveQuartic(
    double a, double b, double c, double d, double e,
    double roots[4]);
```
Uses `double` internally for numerical stability; converts back to `float` for `t`.

### Normal

```
P'     = P − center          // world hit point relative to torus center
denom  = sqrt(P'.x² + P'.z²)
factor = 1.0f − R / denom    // R = major_radius
N      = normalize(P'.x · factor, P'.y, P'.z · factor)
```

Guard: if `denom < ε`, fall back to `(0, sign(P'.y), 0)`.

### Entry points required

Same pattern as cone: `__intersection__torus`, `__closesthit__torus`,
`__closesthit__torus_shadow`, `__anyhit__torus_shadow`.

## Program Groups (`PipelineManager`)

Six new program groups, mirroring the cylinder pattern exactly:

| Group | Type | CH | AH | IS |
|---|---|---|---|---|
| `cone_hitgroup_prog_group` | radiance | `__closesthit__cone` | — | `__intersection__cone` |
| `cone_shadow_hitgroup_prog_group` | shadow | `__closesthit__cone_shadow` | `__anyhit__cone_shadow` | `__intersection__cone` |
| `photon_cone_hitgroup` | photon | `__closesthit__photon` | — | `__intersection__cone` |
| `torus_hitgroup_prog_group` | radiance | `__closesthit__torus` | — | `__intersection__torus` |
| `torus_shadow_hitgroup_prog_group` | shadow | `__closesthit__torus_shadow` | `__anyhit__torus_shadow` | `__intersection__torus` |
| `photon_torus_hitgroup` | photon | `__closesthit__photon` | — | `__intersection__torus` |

SBT offsets: `STRIDE_RAY_TYPES * GEOMETRY_TYPE_CONE` and `STRIDE_RAY_TYPES * GEOMETRY_TYPE_TORUS`.

All six destroyed in `PipelineManager::destroy()`.

## Scala JNI Layer (`OptiXRenderer.scala`)

Two new public methods following the cylinder API convention:

```scala
def addConeInstance(
  apex: Vector[3], base: Vector[3], radius: Float, material: Material
): Option[Int]

def addTorusInstance(
  center: Vector[3], axis: Vector[3],
  majorRadius: Float, minorRadius: Float,
  material: Material
): Option[Int]
```

Each backed by a `@native` method. The native implementation stages the data into the device buffer
and adds an instance with the correct `GEOMETRY_TYPE_CONE` / `GEOMETRY_TYPE_TORUS` value.

## Scene Builders

### `ConeSceneBuilder`

- Parses `ObjectSpec` fields: `center` (→ base center), `apex`, `radius`, material fields.
- Calls `renderer.addConeInstance(...)`.
- Registered in `GeometryRegistry` for `ObjectType "cone"`.

### `TorusSceneBuilder`

- Parses `ObjectSpec` fields: `center`, `major-radius`, `minor-radius`, material fields.
- `axis` defaults to `(0,1,0)` (Y-axis); field accepted but Sprint 19 validates `axis ≈ (0,1,0)`.
- Calls `renderer.addTorusInstance(...)`.
- Registered in `GeometryRegistry` for `ObjectType "torus"`.

## CLI Syntax

```
--objects type=cone:apex=0,1,0:center=0,0,0:radius=0.5:color=0.8,0.2,0.1:roughness=0.3
--objects type=torus:center=0,0,0:major-radius=1.0:minor-radius=0.3:color=0.2,0.6,0.9
```

`center` for cones = base center (consistent with sphere/cylinder naming convention).

## `ObjectType` additions

Add `"cone"` and `"torus"` to `VALID_TYPES` (14 → 16 entries). No new named set constant needed;
`GeometryRegistry` routes each type to its builder, and `VALID_TYPES` is the single validation
source of truth.

## Constraints and Non-Goals

- No plane implementation (Task 19.4 is separate).
- Torus axis generalization deferred to Sprint 20.
- No UV coordinates on cone/torus in Sprint 19 (texture mapping is a future task).
- `solveQuartic` is a local `__device__` function in `hit_torus.cu`; no shared solver library.
